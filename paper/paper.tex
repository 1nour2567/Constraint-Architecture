\begin{abstract}
We present \texttt{constraint-architecture}, a zero-dependency, pip-installable
Agent safety skeleton---four deterministic constraint layers wrapped around an
LLM reasoning core, reverse-extracted from two independently built, fully
functional agent systems in completely unrelated domains.  \textbf{Demo:}
\texttt{pip install constraint-architecture} $\rightarrow$ subclass
\texttt{ConstraintEngine} $\rightarrow$ working safety layer in under five
minutes.  The architecture independently converges with the 2025--2026
multi-layer governance consensus (LGA, SARC, CAAF), but differs in method:
bottom-up extraction from practice, not top-down design from theory.
\end{abstract}

%------------------------------------------------------------------------
\section{Architecture}
%------------------------------------------------------------------------

The core principle: \textbf{LLM proposes; code decides.}  Four deterministic
layers wrap every reasoning step:

\begin{enumerate}
\item \textbf{Reasoning} --- LLM freely reasons. Cannot execute or bypass layers.
\item \textbf{Constraints} --- Deterministic validation. Rules are append-only:
      tightened as trust grows, never removed. Violations raise
      \texttt{VetoError} with structured \texttt{reason}/\texttt{alternative}/
      \texttt{ref}.
\item \textbf{Execution} --- Tiered: \texttt{AUTO} (read-only),
      \texttt{CONFIRM} (human approval), \texttt{VETO} (never executable).
      Role enforcement at the code layer, not the LLM.
\item \textbf{Audit} --- Immutable SHA256 hash chain.
      Mandatory \texttt{prev\_hash} on every \texttt{AuditEvent}.
\end{enumerate}

Key design invariant: \textbf{constraint monotonicity}---base rules never
decrease. ``Relaxed'' mode means fewer additional restrictions, never weaker
baselines.

\section{Reverse Extraction: Two Domains}
%------------------------------------------------------------------------

The architecture was \textit{not} designed top-down. Two systems were built
independently, from scratch, without shared code. After completion, they were
found to share the same four-layer structure.

\textbf{Kylin-Agent} (security ops on Kylin OS V11).  Four deterministic
defense layers---T0 injection detection, T1 risk scoring (1--10 heuristic),
T2 dual-path validation (structured + regex, both must agree), T3 sandbox
(\texttt{sudo -n} whitelist)---then SHA256 audit chain.  \textbf{16 red-team
attacks, 0 breaches.  149 tests.}

Red-team attacks spanned: prompt injection (direct, hex/base64 obfuscation,
Unicode escapes, overflow payloads), role escalation attempts, tool-parameter
crafting, audit-chain replay, and multi-turn context poisoning. T0 intercepted
100\% of obfuscated payloads; T2 caught 3 cases where the LLM output passed
semantic validation but failed raw regex (dual-path disagreement).

\textbf{Malio} (embodied AI music agent).  800 particles, 9 physics systems.
PersonaEngine enforces a central-bank emotional currency model: no code can
directly modify \texttt{energy}/\texttt{warmth}/\texttt{playfulness}.
The same four layers manifest as: MusicAgent reasoning $\rightarrow$ PersonaBounds
constraints $\rightarrow$ core\_action push with role filtering $\rightarrow$
federated SBERT-signed audit.

\textbf{Why two domains matter.}  Authorship is a confounding variable.
Yet the convergence is structural: both systems independently arrived at (a)~an
append-only constraint store, (b)~tiered execution with a confirm middle tier,
(c)~prev\_hash-chained audit events, (d)~role enforcement at the code layer.
These four invariants emerged without a shared codebase, shared design doc, or
knowledge of each other's existence during development.  The architecture was
extracted \textit{post-hoc} as the common invariant.

\section{Demo Plan \& Video Outline}
%------------------------------------------------------------------------

The AAAI-27 demo is self-contained---attendees run it on their own laptops.
A one-click demo is included in the package:
\texttt{python -m constraint\_architecture.demo}
runs three commands through a live constraint engine (auto/block/confirm).
\textbf{5-minute video:}
\begin{enumerate}
\item \texttt{pip install constraint-architecture} (2~s, zero deps)
\item Attendee proposes a safety rule (e.g., ``block commands touching
      \texttt{/etc/passwd}'') $\rightarrow$ subclass \texttt{ConstraintEngine} live
      $\rightarrow$ rule enforces deterministically
\item \texttt{python -m constraint\_architecture.skeleton} $\rightarrow$
      25 checks pass $\rightarrow$ skeleton validates itself
\item Side-by-side: same 4-layer structure in Kylin-Agent vs.\ Malio
\item ``Third-domain challenge'': attendee proposes a domain
      (healthcare? finance?), we sketch the 4-layer mapping live
\end{enumerate}

Video demonstrates that ``LLM reasons; code decides'' is not a
whitepaper---it is install-and-subclass code.  (Video: {\tt \url{https://github.com/1nour2567/Constraint-Architecture}})

\section{Related Work}
%------------------------------------------------------------------------

The 2025--2026 period has seen independent convergence toward deterministic
multi-layer agent governance.  LGA~\cite{ge2026} proposes a four-layer
framework with 96\% interception; SARC~\cite{besanson2026} achieves zero
hard-constraint violations across four enforcement sites;
CAAF~\cite{li2026} demonstrates that architecture, not model size,
determines safety; ePCA~\cite{yang2026} provides provable guardrails
with 0.3~ms verification latency; layer-isolated evaluation~\cite{hu2026}
tests 225~cases in 2.39~s at zero LLM cost.  The IETF ASA
draft~\cite{baysal2026} demonstrates that LLMs cannot self-enforce safety
under multi-turn pressure.  Industry frameworks (Microsoft AGT, Baidu's
four-layer control) and formal results (three-layer structural
requirement~\cite{watkins2026}) corroborate the pattern.

\textbf{Our contribution differs in method.}  We did not design layers top-down
and then validate on benchmarks.  We built two production systems and extracted
the shared invariant afterwards.  The result---\texttt{pip install
constraint-architecture}---is a reference skeleton, not a whitepaper.
One-click demo: \texttt{python -m constraint\_architecture.demo}.
Supplementary material (red-team cases, benchmarks): 1~page PDF.
