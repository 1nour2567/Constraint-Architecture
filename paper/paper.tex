\begin{abstract}
Recent work (LGA, SARC, CAAF, ePCA) establishes multi-layer deterministic
governance as the emerging standard for agent safety.  We build on this
direction with three contributions specific to \textit{reverse extraction from
practice}: (1)~Cross-domain structural validation---two independently built,
fully functional agent systems in unrelated domains (security ops and embodied
music) converged on the same four-layer architecture and four structural
invariants without shared code or design docs.  (2)~Reverse-extraction
methodology---every existing framework derives layers from theory; we extracted
the invariant \textit{after} discovering it in production.  (3)~An
engineering-grade artifact---\texttt{pip install constraint-architecture}: zero
dependencies, 7~ABCs + 7~Protocols, 10~frozen dataclasses, 25~automated
self-checks.  \textbf{Demo:} \texttt{python -m constraint\_architecture.demo}
demonstrates three commands through a live constraint engine in under one
second, with no configuration.
\end{abstract}

\section{Architecture}
%------------------------------------------------------------------------

Recent work---LGA \cite{ge2026}, SARC \cite{besanson2026}, CAAF \cite{li2026},
ePCA \cite{yang2026}, and the IETF ASA draft \cite{baysal2026}---has established
that multi-layer deterministic governance is the right direction for agent
safety.  We build on this direction but make three contributions specific to
\textit{reverse extraction from practice}:

\begin{enumerate}
\item \textbf{Cross-domain structural validation.}
Two fully functional agent systems---Kylin-Agent (security ops on Kylin OS V11,
16 red-team attacks, 0 breaches) and Malio (embodied AI music, 800 particles,
9 physics systems)---were built independently, without shared code or design
docs.  After completion, both were found to share the same four-layer structure
and four structural invariants: append-only constraint store, tiered execution
with a confirm middle tier, prev\_hash-chained audit events, and role
enforcement at the code layer.
\item \textbf{Reverse-extraction methodology.}
Every existing framework derives layers from theory and validates on benchmarks.
We are the only work that extracted the invariant \textit{after} building
production systems.  If the same architecture emerges from unrelated domains
that share no common ancestor, it is likely capturing a real engineering
invariant rather than a design preference.
\item \textbf{Engineering-grade artifact, not a whitepaper.}
The result is \texttt{pip install constraint-architecture}: zero dependencies,
1384~lines of Python, 7~ABCs + 7~Protocols, 10~frozen dataclasses with
\texttt{\_\_post\_init\_\_} validation, 4~custom exceptions with structured
fields, and 25~automated self-checks.  Architectural commitments (immutability,
monotonic constraints, mandatory audit prev\_hash) are enforced at the code
level---not documented in a figure.
\end{enumerate}

Key design invariant: \textbf{constraint monotonicity}---base rules never
decrease. ``Relaxed'' mode means fewer additional restrictions, never weaker
baselines.

\section{Reverse Extraction: Two Domains}
%------------------------------------------------------------------------

The architecture was \textit{not} designed top-down. Two systems were built
independently, from scratch, without shared code. After completion, they were
found to share the same four-layer structure.

\textbf{Kylin-Agent} (security ops on Kylin OS V11).  Four deterministic
defense layers---T0 injection detection, T1 risk scoring (1--10 heuristic),
T2 dual-path validation (structured + regex, both must agree), T3 sandbox
(\texttt{sudo -n} whitelist)---then SHA256 audit chain.  \textbf{16 red-team
attacks, 0 breaches.  149 tests.}

Red-team attacks spanned: prompt injection (direct, hex/base64 obfuscation,
Unicode escapes, overflow payloads), role escalation attempts, tool-parameter
crafting, audit-chain replay, and multi-turn context poisoning. T0 intercepted
100\% of obfuscated payloads; T2 caught 3 cases where the LLM output passed
semantic validation but failed raw regex (dual-path disagreement).

\textbf{Malio} (embodied AI music agent).  800 particles, 9 physics systems.
PersonaEngine enforces a central-bank emotional currency model: no code can
directly modify \texttt{energy}/\texttt{warmth}/\texttt{playfulness}.
The same four layers manifest as: MusicAgent reasoning $\rightarrow$ PersonaBounds
constraints $\rightarrow$ core\_action push with role filtering $\rightarrow$
federated SBERT-signed audit.  Full attack taxonomy and benchmarks are provided
as 1-page supplementary material.

\section{Demo Plan \& Video Outline}
%------------------------------------------------------------------------

The AAAI-27 demo is self-contained---attendees run it on their own laptops.
\texttt{python -m constraint\_architecture.demo} runs three commands through
a live constraint engine (auto/block/confirm), zero config.
\textbf{5-minute video:}
\begin{enumerate}
\item \texttt{pip install constraint-architecture} (2~s, zero deps)
\item Attendee proposes a safety rule (e.g., ``block commands touching
      \texttt{/etc/passwd}'') $\rightarrow$ subclass \texttt{ConstraintEngine} live
      $\rightarrow$ rule enforces deterministically
\item \texttt{python -m constraint\_architecture.skeleton} $\rightarrow$
      25 checks pass $\rightarrow$ skeleton validates itself
\item Side-by-side: same 4-layer structure in Kylin-Agent vs.\ Malio
\item ``Third-domain challenge'': attendee proposes a domain
      (healthcare? finance?), we sketch the 4-layer mapping live
\end{enumerate}

Video demonstrates that ``LLM reasons; code decides'' is not a
whitepaper---it is install-and-subclass code.  (Video: {\tt \url{https://github.com/1nour2567/Constraint-Architecture}})

\section{Related Work}
%------------------------------------------------------------------------

Recent work---LGA \cite{ge2026}, SARC \cite{besanson2026}, CAAF \cite{li2026},
ePCA \cite{yang2026}, ShieldAgent \cite{zhao2025}, and the IETF ASA
draft \cite{baysal2026}---proposes layered architectures for agent safety.
We explicitly build on this direction but note three gaps that our work fills:

\textbf{Domain-specific constraint engineering.}
Existing frameworks validate safety via benchmark interception rates.
Our constraints---append-only store, \texttt{AUTO/CONFIRM/VETO} tiering,
mandatory \texttt{prev\_hash} chaining, immutable frozen dataclasses with
\texttt{\_\_post\_init\_\_} value-bound enforcement---are engineering
invariants extracted from live systems, not a priori specification.

\textbf{Cross-domain structural validation.}
The four structural invariants named above emerged independently in two
unrelated production systems (security ops and embodied music) that shared
no code, design doc, or development knowledge.  No existing framework reports
this form of convergent-evolution validation from within a single codebase.

\textbf{Engineering-grade artifact, not policy document.}
The result of our extraction is \texttt{pip install constraint-architecture}
(v1.5.1): zero dependencies, 25 automated self-checks, and a one-click demo
(\texttt{python -m constraint\_architecture.demo}).  Architectural commitments
are enforced at the code level---frozen dataclasses prevent veto flipping,
\texttt{\_\_post\_init\_\_} bounds \texttt{risk\_score} to 0--9,
\texttt{AuditEvent} requires \texttt{prev\_hash} at construction.
Layer-isolated evaluation \cite{hu2026} provides the testing methodology;
our skeleton provides the structural guarantee that makes such isolation
meaningful.
